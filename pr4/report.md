ПР №4. Аудит событий: journalctl и auditd
1. journalctl — системный журнал
Что зафиксировал журнал о команде sudo cat /etc/shadow
Май 08 00:45:08 debian sudo[1337]: fulldisnored : TTY=pts/0 ; PWD=/root ; USER=root ; COMMAND=/bin/cat /etc/shadow

Разбор полей:

Май 08 00:45:08 — дата и время события
debian — имя хоста
sudo[1337] — процесс и его идентификатор
fulldisnored — пользователь, выполнивший команду
TTY=pts/0 — терминал, с которого выполнена команда
PWD=/root — рабочая директория
USER=root — пользователь, от чьего имени выполнена команда
COMMAND=/bin/cat /etc/shadow — сама команда

Ошибки в системе с последней загрузки
В системе наблюдаются ошибки spice-vagent, связанные с невозможностью подключения к org.gnome.Mutter.DisplayConfig. Это ошибки графической подсистемы, которые не влияют на безопасность системы и работу аудита.

Статистика входов
Успешных входов: 0
Неудачных попыток: 0

SSH-сервер не использовался, поэтому попытки входа через ssh отсутствуют.

2. auditd — настройка правил
Применённые правила
-w /etc/passwd -p rwa -k auth-files
-w /etc/shadow -p rwa -k auth-files
-w /etc/group -p rwa -k auth-files
-w /etc/sudoers -p rwa -k priv-files
-w /etc/sudoers.d -p rwa -k priv-files
-w /etc/ssh/sshd_config -p rwa -k ssh-config
-a always,exit -S open -S openat -F success=0 -k access-denied
-a always,exit -S unlink -S unlinkat -k file-delete
-a always,exit -S execve -F euid=0 -F auid!=0 -F auid!=-1 -k priv-exec
-b 8192

Разбор записи SYSCALL
Поле auid — 0 — исходный идентификатор пользователя, который вошёл в систему. Это значение не меняется даже при использовании sudo.

Поле uid — 0 — текущий идентификатор пользователя. Значение 0 означает пользователя root.

Поле euid — 0 — эффективный идентификатор пользователя, используемый для проверки прав доступа.

Поле comm — cat — имя команды или процесса, который выполнил системный вызов.

Поле exe — /bin/cat — полный путь к исполняемому файлу.

3. Расследование инцидента
Хронология действий bob
Действие	Результат	Источник
cat /etc/shadow	Отказ в доступе	ausearch -k auth-files
ls /root	Отказ в доступе	ausearch -k access-denied
cat /etc/passwd	Успешно	ausearch -k auth-files
touch /tmp/totally-not-malware.sh	Успешно	ausearch --start recent
cat /etc/sudoers	Отказ в доступе	ausearch -k priv-files
Как auditd помог расследованию
Auditd зафиксировал все попытки пользователя bob получить доступ к защищённым файлам. Даже попытки, которые завершились отказом, были записаны в журнал. Был обнаружен подозрительный файл totally-not-malware.sh, созданный bob. С помощью ключей auth-files и priv-files удалось отследить все обращения к файлам аутентификации и конфигурации sudo.

Чем auditd лучше journalctl для задач аудита безопасности
Auditd работает на уровне ядра операционной системы и перехватывает системные вызовы. Это означает, что любое приложение не может скрыть свои действия, даже если попытается обойти системное логирование. Journalctl получает только те сообщения, которые приложения сами решают отправить через syslog. Кроме того, auditd фиксирует неудачные попытки доступа, что критически важно для расследования инцидентов информационной безопасности.
4. Связь с нормативкой
Мера АУД.4 Аудит безопасности — реализована через настройку auditd с правилами мониторинга файлов /etc/passwd, /etc/shadow, /etc/sudoers.

Мера АУД.7 Мониторинг несанкционированного доступа — реализована через фиксацию неудачных попыток доступа bob к файлам /etc/shadow и /root с помощью правила access-denied.

Выводы
В ходе выполнения практической работы были получены следующие навыки:

Работа с journalctl — фильтрация системных журналов по времени, службам, уровню ошибок и поиск событий аутентификации.

Настройка auditd — создание временных и постоянных правил для слежения за критичными файлами системы.

Расследование инцидента — поиск действий пользователя через ausearch с использованием ключей auth-files, priv-files и access-denied.

Анализ SYSCALL-записей — понимание полей auid, uid, euid, comm и exe для идентификации пользователя и его действий.

Auditd является более мощным инструментом для задач информационной безопасности по сравнению с journalctl, так как работает на уровне ядра и не может быть обойдён приложением, а также фиксирует все попытки доступа, включая неудачные.
