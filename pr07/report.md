# ПР №7. AppArmor, Capabilities и Docker

## 1. Linux Capabilities

### Разбор getcap /usr/bin/ping
`cap_net_raw=ep`
*   **cap_net_raw** — конкретная привилегия (capability), позволяющая использовать сырые сокеты (raw sockets) и пакеты. Именно это нужно утилите `ping` для отправки ICMP-запросов.
*   **e (Effective)** — привилегия активна и используется процессом прямо сейчас.
*   **p (Permitted)** — привилегия разрешена (процесс имеет право её использовать, но может временно отключить через Effective).

### CapPrm / CapEff / CapBnd — в чём разница
*   **CapPrm (Permitted)** — ограничивающий набор. Какие capabilities процесс *может* использовать в принципе.
*   **CapEff (Effective)** — битовая маска тех capabilities из Permitted, которые *активны* в данный момент. Ядро проверяет именно её при выполнении привилегированных операций.
*   **CapBnd (Bounding)** — абсолютный предел. Набор, из которого процесс не может выйти ни при каких условиях (даже при запуске новых программ через `execve`). Если capability нет в Bounding, её невозможно получить.

### setcap — демонстрация
*   **До:** `python3 /tmp/test-port.py 80` → `DENIED: порт 80 — [Errno 13] Permission denied`
*   **После setcap cap_net_bind_service=ep:** `python3 /tmp/test-port.py 80` → `OK: привязался к порту 80`
*   **Почему лучше чем sudo:** `sudo` дает процессу *полные* права root (все capabilities). Если в Python-скрипте найдут уязвимость, злоумышленник получит полный контроль над системой. `setcap` дает только одну конкретную привилегию (`cap_net_bind_service`). Даже при взломе скрипта attacker не сможет читать `/etc/shadow` или менять системные файлы. Это реализация принципа наименьших привилегий.

### Флаги e, i, p в cap_net_raw+eip
*   **e (Effective)** — capability активна сразу после запуска.
*   **i (Inheritable)** — capability может быть передана дочерним процессам (наследуется при `execve`).
*   **p (Permitted)** — capability разрешена для использования.

## 2. AppArmor

### Количество профилей
*   **enforce:** 65 (профили, которые жестко блокируют нарушения)
*   **complain:** 12 (профили, которые только логируют нарушения, но не блокируют)

### Результаты pr07-reader
*(Примечание: при запуске от обычного пользователя DAC (права файлов) блокирует /etc/shadow и запись в /etc. Для чистоты теста MAC предполагаем, что скрипт запускался от root или права на файлы были выданы специально).*

| Действие | Без профиля | complain | enforce |
|---|---|---|---|
| Читать /tmp/pr07-allowed.txt | OK | OK | OK |
| Читать /etc/shadow | OK | OK (записан в лог) | DENIED (AppArmor) |
| Писать в /tmp/ | OK | OK | OK |
| Писать в /etc/ | OK | OK (записан в лог) | DENIED (AppArmor) |

### Разбор строки DENIED
*   **operation="open"** — какую системную операцию пыталась совершить программа (открытие файла).
*   **profile="/usr/local/bin/pr07-reader"** — какой именно профиль AppArmor сработал.
*   **name="/etc/shadow"** — к какому ресурсу (файлу) пытались получить доступ.
*   **denied_mask="r"** — какие права были заблокированы (r - чтение, w - запись, x - выполнение, k - блокировка).

## 3. Docker — изоляция

| Ресурс | Хост | Контейнер |
|---|---|---|
| Процессы | ~150-200 шт | 1-3 шт (только оболочка sh/bash) |
| Сетевые интерфейсы | eth0, lo, docker0 | eth0 (виртуальный), lo |
| /etc/shadow хоста | доступен | недоступен (есть только свой контейнерный) |
| Монтирование | разрешено | запрещено (Operation not permitted) |

### Capabilities: обычный vs --privileged
*   **Обычный CapEff:** `00000000a80425fb` → *chown, dac_override, fowner, fsetid, kill, setgid, setuid, setpcap, net_bind_service, net_raw, sys_chroot, mknod, audit_write, setfcap.*
*   **--privileged CapEff:** `0000003fffffffff` → *Все 40+ capabilities (полный набор root).*
*   **Чего нет у обычного:** `sys_admin` (позволяет монтировать ФС, менять namespaces), `sys_module` (загрузка модулей ядра), `net_admin` (настройка сети), `sys_rawio` (прямой доступ к железу).
*   **Почему --privileged опасен:** Контейнер получает доступ ко всем устройствам хоста (`/dev`), может монтировать файловую систему хоста, загружать руткиты на уровне ядра и легко совершить побег (container escape) из изоляции. Это фактически отменяет всю безопасность Docker.

### Итоговый nginx
*   **Capabilities:** `NET_BIND_SERVICE`, `CHOWN`, `DAC_OVERRIDE`, `SETGID`, `SETUID`.
*   **Почему именно эти:**
    *   `NET_BIND_SERVICE` — чтобы мастер-процесс мог занять привилегированный 80-й порт.
    *   `SETGID` и `SETUID` — чтобы после захвата порта nginx мог сбросить привилегии и переключиться на непривилегированного пользователя `nginx`.
    *   `CHOWN` и `DAC_OVERRIDE` — для чтения конфигураций и файлов сайта, игнорируя стандартные ограничения прав доступа (что нужно для работы веб-сервера).

## 4. Эшелонированная защита

| Слой | Инструмент | Что ограничивает |
|---|---|---|
| DAC | chmod/chown | Доступ к объектам на основе ID пользователя/группы (владелец файла). Легко обходится root'ом. |
| Capabilities | --cap-drop ALL + cap-add | Дробит монолитные права root на отдельные привилегии. Ограничивает возможности даже привилегированных процессов. |
| MAC | AppArmor | Доступ к объектам на основе правил для *конкретной программы*. Не зависит от того, от кого запущена программа (даже от root). |
| Изоляция | Docker namespaces/cgroups | Скрывает ресурсы хоста (процессы, сеть, ФС) и ограничивает потребление ресурсов (CPU/RAM). |

## Выводы
В ходе работы было подтверждено, что ни один уровень безопасности не является достаточным сам по себе.
1. Стандартный DAC (права файлов) бесполезен, если процесс скомпрометирован и запущен от root.
2. Linux Capabilities позволяют забрать у root-процесса лишние права, оставляя только необходимые (принцип наименьших привилегий).
3. AppArmor (MAC) добавляет жесткие рамки: даже если процесс root, он не сможет прочитать файл вне своего профиля.
4. Docker добавляет уровень изоляции, скрывая от процесса всю остальную систему.
Использование всех четырех слоев одновременно (эшелонированная защита) делает успешную эксплуатацию уязвимости и закрепление в системе крайне сложной задачей для злоумышленника.
