# ПР №6. Анализ сетевого трафика: tcpdump и Wireshark

## 1. Разбор строки tcpdump

Строка: `12:34:56.789012 IP 127.0.0.1.44321 > 127.0.0.1.8080: Flags [S], seq 123, length 0`

| Поле | Значение | Что означает |
|------|----------|--------------|
| 12:34:56.789012 | 12:34:56.789012 | Время захвата пакета с точностью до микросекунд |
| IP | Internet Protocol | Протокол сетевого уровня (IPv4) |
| 127.0.0.1.44321 | 127.0.0.1:44321 | IP-адрес и порт источника (loopback, порт 44321) |
| > | направление | Указывает направление передачи от источника к получателю |
| 127.0.0.1.8080 | 127.0.0.1:8080 | IP-адрес и порт назначения (loopback, порт 8080) |
| Flags [S] | SYN | Флаг TCP SYN — запрос на установление соединения |
| seq 123 | Sequence Number 123 | Начальный порядковый номер TCP-сегмента |
| length 0 | 0 байт | Длина полезной нагрузки (нет данных, только заголовок) |

## 2. Перехваченные данные HTTP


Что видно в трафике: при использовании протокола HTTP все передаваемые данные, включая логин и пароль, передаются в открытом текстовом виде. В выводе tcpdump можно явно увидеть строки `username=admin&password=supersecret123`, которые были отправлены через POST-запрос.

Кто в реальной сети может так же прочитать этот трафик:
Злоумышленник, находящийся в той же локальной сети (например, подключённый к тому же Wi-Fi)
Администратор сети, имеющий доступ к маршрутизаторам или коммутаторам
Провайдер или любой оператор связи на пути передачи пакетов
Владелец вредоносного ПО, установленного на любом узле сети

## 3. Follow TCP Stream

Что видит злоумышленник, перехвативший этот трафик: злоумышленник видит полный диалог между клиентом и сервером. В окне Follow TCP Stream отображаются:
- Красным цветом — данные, отправленные клиентом (включая логин, пароль, заголовки HTTP)
- Синим цветом — ответ сервера

Таким образом, злоумышленник может восстановить полную картину взаимодействия, включая учётные данные и содержимое запросов.

## 4. HTTP vs HTTPS — сравнение

| Что наблюдаем | HTTP (порт 8080) | HTTPS (порт 443) |
|---------------|------------------|------------------|
| IP-адреса сторон | Видны | Видны |
| Порт назначения | Виден (8080) | Виден (443) |
| Заголовки HTTP | Видны открытым текстом | Зашифрованы, не видны |
| Содержимое запроса (body) | Видно открытым текстом | Зашифровано, не видно |
| Логин и пароль | Видны открытым текстом | Зашифрованы, не видны |
| Имя домена (SNI) | Видно в заголовке Host | Видно в SNI (TLS), но без параметров запроса |

## 5. DNS и приватность

Какие домены видны в DNS даже при HTTPS: даже при использовании HTTPS, DNS-запросы передаются в открытом виде. В перехваченном трафике видны домены: github.com, example.com, google.com.

Что это означает для приватности: злоумышленник, перехватывающий трафик, может узнать:
Какие сайты посещает пользователь (даже если содержимое зашифровано)
Время и частоту посещений
Хотя содержимое запросов остаётся скрытым, сам факт обращения к конкретным доменам является утечкой информации.

Для решения этой проблемы используются DNS over HTTPS (DoH) или DNS over TLS (DoT), которые шифруют DNS-запросы.

## 6. Base64

Декодированная строка: `dmFzeWE6TXlQYXNzd29yZDIwMjQ=` → `vasya:MyPassword2024`

Почему Base64 не является шифрованием: Base64 — это схема кодирования (encoding), а не шифрования (encryption). Она преобразует бинарные данные в текстовый формат для безопасной передачи, но не использует ключ и не обеспечивает конфиденциальность. Любой, кто получит строку в Base64, может легко её декодировать с помощью стандартных инструментов. Настоящее шифрование требует использования ключа (симметричного или асимметричного) и обеспечивает защиту от чтения без знания этого ключа.

## 7. tshark vs tcpdump

| Инструмент | Когда использовать |
|------------|--------------------|
| tcpdump | Быстрый захват трафика в командной строке, минималистичный, работает на любых системах (включая встраиваемые), удобен для сбора пакетов для последующего анализа |
| tshark | Анализ pcap-файлов без графического интерфейса, применение display-фильтров, получение статистики (io,phs), работа на серверах без GUI |
| Wireshark | Детальный анализ с графическим интерфейсом, визуализация потоков (Follow Stream), удобство интерпретации данных, работа с пакетами в режиме реального времени |

## 8. Каналы утечки через сеть

Какие меры защиты закрывают сетевой канал утечки:
 Использование HTTPS вместо HTTP для шифрования данных
 Использование DNS over HTTPS (DoH) или DNS over TLS (DoT) для защиты DNS-запросов
 Использование шифрования на уровне приложений (PGP, S/MIME для почты)
 Настройка брандмауэров и систем обнаружения вторжений (IDS)
 Применение сетевых политик и сегментации сети
- Использование TLS 1.3 с шифрованием сертификатов и защитой SNI

## Выводы

В ходе практической работы освоены инструменты анализа сетевого трафика tcpdump, tshark и Wireshark. Выполнен захват и анализ HTTP-трафика, в результате которого наглядно продемонстрировано, что логин и пароль передаются в открытом виде. Проведено сравнение HTTP и HTTPS: показано, что HTTPS шифрует заголовки, тело запроса и учётные данные, но IP-адреса и имя домена (SNI) остаются видимыми. Обнаружено, что даже при использовании HTTPS DNS-запросы передаются в открытом виде, раскрывая посещаемые сайты. Изучено, что Base64 является кодированием, а не шифрованием. Сформулированы меры защиты сетевого канала утечки информации: переход на HTTPS, использование DoH/DoT, VPN и шифрование на уровне приложений.
