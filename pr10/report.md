# ПР №10. Лицензирование и open-source compliance

## 1. Выбор лицензии

**Выбранная лицензия:** MIT
**Тип:** разрешительная (permissive)

**Что разрешает:** использовать, копировать, модифицировать, продавать код, использовать в закрытых проектах.
**Что требует от пользователей кода:** сохранять уведомление об авторских правах и текст лицензии во всех копиях.
**Может ли кто-то продавать ваш код:** Да, может.

## 2. Результат licensee detect

Вывод команды: `licensee` не удалось установить в системе, но инструмент сканирует репозиторий и сравнивает текст LICENSE с базой известных лицензий.
Что означает Match %: процент совпадения текста вашей лицензии с эталонным текстом. GitHub использует похожий алгоритм ( licensee ) для автоматического определения лицензии репозитория (показывает бейджик лицензии в правой колонке).

## 3. SPDX-заголовки

Пример заголовка, который добавили в файл pr09/report.md: SPDX-FileCopyrightText: 2026 ЕНИН АНДРЕЙ <aei131205@mail.ru>
SPDX-License-Identifier: MIT
Что означает каждая строка:
- SPDX-FileCopyrightText: указывает владельца авторских прав и год.
- SPDX-License-Identifier: указывает идентификатор лицензии (MIT) в стандартизированном формате SPDX.

Результат reuse lint:
Инструмент показал, что проект не полностью соответствует REUSE Specification 3.3, так как из 31 файла только у 1 (pr09/report.md) есть явные заголовки. Остальные файлы (картинки, старые отчеты) не имеют SPDX-тегов. Для полного соответствия нужно либо добавить теги ко всем файлам, либо создать файл `.reuse/dep5` с массовым указанием лицензий.

## 4. Лицензии зависимостей

Таблица из pip-licenses (основные зависимости):
- requests: Apache-2.0
- flask: BSD-3-Clause
- numpy: BSD-3-Clause
- Werkzeug: BSD-3-Clause

Есть ли GPL/AGPL зависимости: Нет, среди установленных пакетов копилефтных лицензий не обнаружено.
Что это означало бы для коммерческого проекта: Если бы мы использовали GPL-библиотеку, нам пришлось бы открыть исходный код всего нашего коммерческого продукта под GPL, что обычно неприемлемо для бизнеса.

## 5. CI-проверка

Созданы два workflow в `.github/workflows/`:
1. `license-check.yml` — проверяет наличие файла LICENSE и запускает `reuse lint`.
2. `dependency-licenses.yml` — проверяет лицензии зависимостей из `requirements.txt`.

Что произойдёт если добавить GPL-зависимость: Workflow `dependency-licenses.yml` выведет предупреждение "WARNING: GPL/AGPL licensed dependencies found! Review required for commercial use", что позволит разработчикам заметить проблему до слияния кода.

## 6. Анализ совместимости

**Сценарий 1 (MIT + GPL в коммерческом):** Нет, нельзя. GPL требует, чтобы весь производный код распространялся под GPL, то есть придется открыть исходники коммерческого продукта.
**Сценарий 2 (Apache-2.0 + MIT):** Да, можно. MIT совместима с Apache-2.0, нужно лишь сохранить уведомление об авторстве MIT-библиотеки.
**Сценарий 3 (MIT без указания авторства):** Да, нарушает. MIT требует сохранения copyright notice во всех копиях и существенных частях ПО.
**Сценарий 4 (AGPL в SaaS):** Нет, нельзя. AGPL специально создана для закрытия "лазейки SaaS" — она требует раскрытия исходников даже при сетевом взаимодействии.
**Сценарий 5 (нет LICENSE):** Нет, нельзя. По умолчанию (ст. 1259 ГК РФ) код защищен авторским правом с момента создания. Без явной лицензии использование кода запрещено.

## 7. Связь с нормативкой

| Аспект | Статья ГК РФ | Как это реализуется |
|--------|-------------|-------------------|
| Авторское право на ПО | ст. 1259 | ПО охраняется как литературное произведение, права возникают в момент создания. |
| Защита от копирования | ст. 1270 | Исключительное право на использование, воспроизведение и распространение. |
| Использование open-source | ст. 1286.1 | Свободная лицензия — это договор присоединения, где автор заранее дает согласие на использование на определенных условиях. |

## Выводы

В ходе работы мы научились явно лицензировать файлы с помощью SPDX-заголовков и инструмента `reuse`. Выяснилось, что даже если в корне есть LICENSE, это не распространяется автоматически на каждый файл — для строгого compliance (REUSE Specification) нужен заголовок в каждом файле. Также мы проверили лицензии зависимостей через `pip-licenses` и настроили CI-проверки, чтобы автоматически отслеживать появление копилефтных (GPL/AGPL) библиотек в проекте.
